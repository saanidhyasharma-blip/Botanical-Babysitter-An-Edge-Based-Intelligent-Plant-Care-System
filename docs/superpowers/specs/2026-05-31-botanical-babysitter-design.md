# Botanical Babysitter 1.0 — Design Specification

## Overview

A smart plant monitoring and autonomous irrigation platform.

**Subsystems:**
1. ESP32 Edge Device — firmware with REST API, sensor reading, relay control
2. Supabase Backend — PostgreSQL database, Row Level Security, Realtime channels
3. Next.js Dashboard — 6-page premium web app with realtime updates
4. Integration Layer — ESP32 → Supabase data bridge (runs as edge function or cron)
5. Intelligence Engines — health score calculator, alert engine, recommendations

**Approach:** Build firmware and dashboard in parallel. The ESP32 posts sensor data to Supabase via HTTP. The dashboard reads from Supabase with realtime subscriptions. No direct ESP32 ↔ Dashboard connection.

## 1. Data Architecture (Supabase)

### Tables

```sql
-- User profiles (extends Supabase Auth)
profiles (
  id          uuid PRIMARY KEY REFERENCES auth.users(id),
  email       text NOT NULL,
  name        text,
  avatar_url  text,
  created_at  timestamptz DEFAULT now(),
  updated_at  timestamptz DEFAULT now()
)

-- Registered devices
devices (
  id               uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id          uuid REFERENCES profiles(id) ON DELETE CASCADE,
  device_name      text NOT NULL DEFAULT 'My Plant',
  device_uid       text UNIQUE NOT NULL,  -- hardware identifier
  firmware_version text,
  status           text DEFAULT 'offline' CHECK (status IN ('online','offline')),
  last_seen        timestamptz,
  created_at       timestamptz DEFAULT now()
)

-- Time-series sensor data
sensor_readings (
  id          bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  device_id   uuid REFERENCES devices(id) ON DELETE CASCADE,
  temperature real,       -- °C
  humidity    real,       -- %
  moisture    real,       -- % (0 = dry, 100 = saturated)
  pump_status boolean DEFAULT false,
  mode        text DEFAULT 'manual' CHECK (mode IN ('auto','manual')),
  created_at  timestamptz DEFAULT now()
)

-- Alerts generated by the system
alerts (
  id          bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  device_id   uuid REFERENCES devices(id) ON DELETE CASCADE,
  severity    text NOT NULL CHECK (severity IN ('critical','warning','info')),
  title       text NOT NULL,
  message     text,
  read        boolean DEFAULT false,
  created_at  timestamptz DEFAULT now()
)

-- Activity events (pump toggled, mode changed, device connected)
events (
  id          bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  device_id   uuid REFERENCES devices(id) ON DELETE CASCADE,
  event_type  text NOT NULL,
  message     text,
  created_at  timestamptz DEFAULT now()
)
```

### Indexes
- `sensor_readings(device_id, created_at DESC)` — fast time-range queries
- `alerts(device_id, created_at DESC)` — alert timeline
- `events(device_id, created_at DESC)` — activity feed

### Row Level Security
- Every table filtered by `user_id` (or `device_id → devices.user_id`)
- `sensor_readings` INSERT allowed via service role key only (ESP32 uses service key)
- Users can only read their own devices, readings, alerts, events

### Realtime
- Enable Realtime on `sensor_readings` and `alerts` tables
- Dashboard subscribes to `sensor_readings` filtered by device_id
- New readings trigger UI updates without polling

## 2. ESP32 Firmware

**Language:** Arduino C++ (Arduino IDE or PlatformIO)

**Hardware Connections:**
- DHT11 → GPIO 4 (temperature + humidity)
- Soil Moisture → Analog GPIO 34
- Relay → GPIO 5 (water pump control)

**REST API Endpoints (served by ESP32's built-in web server):**

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/api/status` | Returns JSON: temperature, humidity, moisture, pump_status, mode, uptime |
| POST | `/api/pump/on` | Turn pump on, returns status |
| POST | `/api/pump/off` | Turn pump off, returns status |
| POST | `/api/mode/auto` | Switch to auto mode |
| POST | `/api/mode/manual` | Switch to manual mode |

**Auto Mode Logic:**
- Read moisture every 10 seconds
- If moisture < configurable threshold (default 30%): activate pump for 5 seconds
- Cooldown period of 60 seconds between activations
- Deactivate pump when moisture > threshold + 10% (hysteresis)

**Supabase Data Push:**
- Every 30 seconds: POST sensor reading to Supabase REST API
- Use Supabase service role key (stored in firmware config)
- Endpoint: `https://<project>.supabase.co/rest/v1/sensor_readings`
- Also update `devices.last_seen` and `devices.status = 'online'`

**Error Handling:**
- WiFi reconnect loop with exponential backoff
- Sensor read failures: skip reading, log error, continue
- HTTP failures: retry up to 3 times with backoff

**CORS:**
- Allow all origins for local development

### Health Score Engine (Client-Side)
```
score = (moisture_score * 0.4) + (temperature_score * 0.3) + (humidity_score * 0.3)

moisture_score:
  optimal range: 40-70% → 100
  30-40% or 70-80% → 70
  20-30% or 80-90% → 40
  <20% or >90% → 10

temperature_score:
  optimal range: 18-28°C → 100
  15-18° or 28-32° → 70
  10-15° or 32-38° → 40
  <10° or >38° → 10

humidity_score:
  optimal range: 40-70% → 100
  30-40% or 70-80% → 70
  20-30% or 80-90% → 40
  <20% or >90% → 10
```

Labels: ≥85 Excellent, ≥70 Healthy, ≥50 Warning, <50 Critical

## 4. ESP32 → Dashboard Data Flow

```
ESP32 reads sensors
  ↓ every 30s
HTTP POST to Supabase REST API (sensor_readings table)
  ↓
Supabase Realtime broadcasts INSERT
  ↓
Dashboard's useRealtimeSubscription receives new row
  ↓
React Query cache updated
  ↓
UI re-renders with new data
```

**Device control flow (user clicks "Pump ON"):**
```
User clicks button in Dashboard
  ↓
Dashboard calls Supabase Edge Function: /functions/v1/device-command
  ↓
Edge Function calls ESP32's REST API: POST /api/pump/on
  ↓
ESP32 activates relay, responds with status
  ↓
Edge Function returns response to Dashboard
  ↓
ESP32's next sensor reading (within 10s) reflects pump_status = true
  ↓
Dashboard sees updated pump_status via Realtime
```

**Why Edge Function for commands?** The ESP32 is on a local network — the Dashboard (hosted on Vercel) can't reach it directly. The Edge Function acts as a proxy. For local development, the dashboard can call ESP32 directly.

**Alternative for v1 (simpler):** Store commands in a `device_commands` table in Supabase. ESP32 polls this table every 5 seconds. Dashboard writes commands; ESP32 reads and executes them. No Edge Function needed. This adds ~5s latency but is much simpler to implement.

**Decision: Use the polling approach for v1.** Add a `device_commands` table:
```sql
device_commands (
  id          bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  device_id   uuid REFERENCES devices(id) ON DELETE CASCADE,
  command     text NOT NULL,  -- 'pump_on', 'pump_off', 'mode_auto', 'mode_manual'
  executed    boolean DEFAULT false,
  created_at  timestamptz DEFAULT now()
)
```
ESP32 polls `GET /rest/v1/device_commands?device_id=eq.X&executed=eq.false&order=created_at.asc&limit=1`, executes command, then PATCHes `executed = true`.

## 7. What This Design Does NOT Include (v1 scope cuts)

- No pH sensor, LDR, grow light, camera support (future hardware)
- No push notifications (browser only — alerts in-app)
- No multi-plant/multi-zone within one device
- No OAuth/social login (email only)
- No data export
- No public API
- No mobile app (responsive web only)
- No AI/ML plant identification

These are all valid v2+ features but would bloat v1 significantly.

## 8. Success Criteria

The project is complete when:
1. ESP32 sends live sensor data to Supabase every 30 seconds
2. Dashboard shows live temperature, humidity, moisture updating in realtime
3. Pump can be turned on/off from the dashboard (via command polling)
4. Auto mode activates pump when moisture drops below threshold
5. All sensor data is stored and queryable
6. Analytics charts show 24h/7d/30d/90d historical data
7. Health score calculates and displays correctly
8. Alerts generate when readings cross thresholds
9. Dashboard is mobile responsive
10. Lighthouse performance score > 90
11. UI feels premium (not a college project)
12. All pages have working navigation
